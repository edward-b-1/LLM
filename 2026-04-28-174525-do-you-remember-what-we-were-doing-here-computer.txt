❯ I read some papers/articles which suggested FPGAs can provide comparable      
  performance for LLM training as to graphics cards like the 5090. Does that    
  sound right to you?                                                           

● That doesn't sound right to me. FPGAs can be competitive for inference on
  specific, carefully optimised models — low latency, good power efficiency —
  but for training they're generally well behind high-end GPUs.

  The RTX 5090 has:
  - ~3352 TFLOPS of FP8 tensor core throughput
  - 32GB GDDR7 at ~1.8TB/s memory bandwidth
  - A mature, heavily optimised software stack (CUDA, cuDNN, PyTorch, Flash
  Attention)

  FPGAs lack dedicated tensor cores, have much lower memory bandwidth, and
  require you to implement training primitives (matmul, backprop, Adam) yourself
   in HDL or HLS. Even well-funded efforts like Microsoft's Project Brainwave
  used FPGAs for inference only.

  The papers you may have seen are likely comparing FPGAs favourably against
  older or smaller GPUs, or in inference contexts, or measuring power efficiency
   (FLOPS/watt) rather than raw throughput. I'd be curious which papers
  specifically — it's possible they're making a narrower claim than it appears.

❯ For an FPGA with 100k logic cells, do you have any idea what kind of order of 
   magnitude of tensor core-like hardware you would be able to configure within 
   it?                                                                          
● I'd treat this as a rough estimate since it depends heavily on the specific
  FPGA and implementation, but the order of magnitude is:

  ~10–100 GFLOPS for training-relevant precision (FP16/FP32)

  Here's the reasoning:

  A 100K LUT FPGA (roughly an Xilinx Artix-7 class device) has around 200–300 
  DSP blocks. Those are the hardened multiply-accumulate units that do the heavy
   lifting. Each DSP block does roughly one MAC per clock cycle. At ~200–450MHz:

  - INT8: ~50–100 GOPS (one MAC per DSP per cycle)
  - FP16/FP32: much worse — a single FP16 multiplier requires 3–5 DSP blocks, so
   maybe 10–50 GFLOPS

  Compare that to the RTX 5090 at ~838 TFLOPS FP16 — roughly 4 orders of 
  magnitude difference.

  The other fundamental problem for training is memory bandwidth — a small FPGA
  has very little on-chip SRAM and limited external memory interfaces, so even
  if you could do the compute, feeding data to the compute units fast enough is
  extremely difficult.

  FPGAs do much better in inference because you can quantise aggressively
  (INT4/INT8), exploit sparsity in custom ways, and pipeline fixed-shape
  operations. For training with full precision and dynamic computation graphs,
  GPUs are essentially purpose-built for the task and FPGAs can't compete.

❯ Tell me some information about the scaling laws of LLMs based on the number   
  of parameters, compute budget and dataset size. We have a model with a fixed  
  number of paremeters. What do the research papers (such as 'Training          
  Compute-Optimal Large Language Models') suggest would be an optimal size of   
  input data in tokens and compute training time?                               

● The two key papers are:

  Kaplan et al. (2020) — OpenAI Scaling Laws
  The original paper. Found that loss scales as power laws with parameters (N),
  tokens (D), and compute (C). Their conclusion was that given a fixed compute
  budget, you should prioritise scaling parameters over data. This led to the
  trend of large, relatively undertrained models like GPT-3.

  Hoffmann et al. (2022) — Chinchilla (DeepMind)
  Overturned Kaplan's conclusion. Showed that GPT-3, Gopher etc. were
  significantly undertrained relative to their size. Their key finding:

  ▎ For compute-optimal training, parameters and tokens should be scaled 
  ▎ equally. The rule of thumb: ~20 tokens per parameter.

  The compute relationship is:
  C ≈ 6 × N × D
  where C is FLOPs, N is parameters, D is tokens. The 6 comes from ~2N FLOPs for
   the forward pass and ~4N for the backward pass.

  ---
  Applied to your model (~117M parameters):

  Chinchilla-optimal token count:
  20 × 117M ≈ 2.34 billion tokens

  At your batch size and context length (batch=16, ctx=1024):
  2.34B / (16 × 1024) ≈ 143,000 steps

  So your English Wikipedia run to 150k steps was almost exactly 
  Chinchilla-optimal — which is a nice coincidence.

  ---
  Important nuance — Chinchilla optimises for compute, not inference:

  Later work (LLaMA, 2023) pointed out that Chinchilla-optimal makes sense if
  compute is your bottleneck. But if you have a fixed small model that you'll
  run inference on many times, it's worth training beyond Chinchilla on more
  tokens — a smaller, overtrained model is cheaper to run than a larger
  compute-optimal one. LLaMA 7B for example was trained on ~1T tokens vs
  Chinchilla's suggested ~140B.

  For your use case — a personal project on fixed hardware — the LLaMA logic
  applies: training longer than 143k steps on the same model is still
  worthwhile, it just has diminishing returns.

❯ So RoPE is the embedding which uses sin and cosines. It looks a bit like an   
  FFT to me. Can you comment on this? I think it is missing the factors of      
  (-1)^n or (i)^n.                                                              

● Good observation — there's a real connection to the DFT, but with some
  important differences.

  What they share:

  Both RoPE and the DFT use sinusoids at multiple frequencies to encode
  information. In RoPE, the frequencies are:

  θ_i = 1 / 10000^(2i/d)

  This gives a geometric sequence of frequencies — high frequency for small i
  (changes rapidly across positions), low frequency for large i (changes
  slowly). This is directly analogous to the DFT basis functions at different
  frequencies k.

  In both cases the idea is the same: a signal (position, or a time-domain
  sample) is projected onto a set of sinusoidal basis functions at different
  frequencies.

  The key difference — RoPE rotates, DFT projects:

  The DFT computes inner products with complex exponentials e^(2πikn/N) — it
  measures how much of each frequency is present in the signal. The result is a
  spectrum.

  RoPE instead applies a rotation to each query/key vector. For position m, it
  rotates each pair of dimensions (x_{2i}, x_{2i+1}) by angle m·θ_i:

  [x_{2i}  ]     [cos(m·θ_i)  -sin(m·θ_i)] [x_{2i}  ]
  [x_{2i+1}]  →  [sin(m·θ_i)   cos(m·θ_i)] [x_{2i+1}]

  This is a 2D rotation matrix, not a projection. The magnitude of the vector is
   preserved — only its direction changes.

  On your point about (-1)^n or i^n:

  The DFT uses complex exponentials e^(2πikn/N) = cos(2πkn/N) + i·sin(2πkn/N).
  The imaginary unit i is what makes the DFT unitary and gives it its
  phase-tracking properties. The (-1)^n terms appear when you shift frequencies
  (e.g. to centre the spectrum).

  RoPE is deliberately real-valued — it avoids complex numbers by working
  directly with 2×2 rotation matrices on pairs of real dimensions. The
  rotate_half function in our code implements exactly this: it constructs the
  [-x2, x1] term that corresponds to multiplying by i in the complex plane. So
  in a sense, the imaginary unit is there — it's just represented geometrically
  as a 90° rotation rather than algebraically as i.

  Why this matters for attention:

  The key insight is that when you compute the dot product q·k after applying
  RoPE, the result depends only on the relative position (m - n), not the
  absolute positions m and n individually. This falls out of the rotation group
  property: rotating q by m and k by n, then taking the dot product, gives the
  same result as rotating by (m - n). That's the property that makes RoPE work
  as a positional encoding — it's baked into the geometry of the dot product
  rather than added as a separate embedding.

❯ Ok so RoPE is actually mathematically equivalent to a rotation in the form of 
   multiplying by the usual complex exponential.                                

● Exactly. If you treat each pair of dimensions (x_{2i}, x_{2i+1}) as the real
  and imaginary parts of a complex number z = x_{2i} + i·x_{2i+1}, then
  multiplying by e^(im·θ_i) gives:

  z · e^(im·θ_i) = (x_{2i} + i·x_{2i+1})(cos(m·θ_i) + i·sin(m·θ_i))
                 = (x_{2i}·cos - x_{2i+1}·sin) + i(x_{2i}·sin + x_{2i+1}·cos)

  Which is precisely the 2×2 rotation matrix applied to (x_{2i}, x_{2i+1}). The
  rotate_half trick in our code is just the mechanical way of doing this in
  real-valued PyTorch tensors without introducing complex dtypes.

  So RoPE is literally just: interpret the embedding dimensions as complex
  numbers, multiply each pair by a position-dependent complex exponential. The
  dot product between two rotated vectors then gives:

  Re(q · k̄) where q is rotated by e^(im·θ) and k by e^(in·θ)
           = Re(q · k̄ · e^(i(m-n)·θ))

  The relative position (m - n) falls out naturally from the conjugate product —
   which is exactly why it works. It's a clean result from complex analysis
  applied to the geometry of dot products.

❯ What is Flash Attention?                                                      

● Flash Attention is a memory-efficient algorithm for computing scaled
  dot-product attention that produces exactly the same result as standard
  attention but uses far less VRAM and runs faster.

  The problem with standard attention:

  For a sequence of length T, standard attention materialises the full T×T
  attention score matrix in VRAM:

  scores = Q @ K^T  # (B, heads, T, T)
  probs  = softmax(scores)
  out    = probs @ V

  At T=1024, batch=16, 12 heads: that's 16 × 12 × 1024 × 1024 × 2 bytes ≈ 400MB
  just for the attention matrix, and it scales as O(T²) in memory.

  What Flash Attention does:

  Instead of computing the full T×T matrix at once, it tiles the computation
  into blocks that fit in the GPU's fast on-chip SRAM (L1 cache), streams
  through them, and computes the softmax incrementally using the log-sum-exp
  trick. The T×T matrix is never fully materialised.

  Result:
  - Memory: O(T) instead of O(T²)
  - Speed: faster despite doing slightly more FLOPs, because SRAM access is much
   faster than VRAM access — memory bandwidth is usually the bottleneck, not
  compute
  - Output: numerically identical to standard attention

  In our code:

  F.scaled_dot_product_attention with is_causal=True — PyTorch automatically
  dispatches to Flash Attention kernels when the hardware supports it, which on
  the 5090 it does. We get it for free without any extra code.

❯ What is the log-sum-exp trick?                                                

● The problem it solves: softmax involves computing e^x for large values of x,
  which overflows to infinity in floating point.

  Naive softmax:
  softmax(x)_i = e^(x_i) / Σ e^(x_j)

  If x_i = 1000, then e^1000 = inf. Overflow.

  The trick:

  Subtract the maximum value before exponentiating:
  softmax(x)_i = e^(x_i - max(x)) / Σ e^(x_j - max(x))

  This is mathematically identical because the max cancels in numerator and
  denominator:
  e^(x_i - m) / Σ e^(x_j - m)  =  (e^x_i / e^m) / (Σ e^x_j / e^m)  =  e^x_i / Σ
  e^x_j

  Now the largest exponent is always e^0 = 1, so nothing overflows.

  The log-sum-exp part:

  The denominator Σ e^(x_j - m) is often computed in log space as:
  log(Σ e^x_j) = m + log(Σ e^(x_j - m))

  This is the "log-sum-exp" — it lets you accumulate the softmax normaliser
  incrementally across blocks without ever materialising the full sum, which is
  exactly what Flash Attention needs to tile the computation across chunks of
  the sequence while still producing a correctly normalised output.

❯ Remind me what the loss mask is for? Is this just so we train by calculating  
  the loss function on the answer part only?                                    

● Yes exactly. In the SQuAD dataset each sample is formatted as:

  Question: What is the capital of France?\nAnswer: Paris

  Without a mask, the loss is computed over every token in the sequence —
  including the question tokens. But we don't want the model to "learn" to
  predict the question, since at inference time the question is given as a
  prompt. We only want it to learn to predict the answer tokens.

  The mask is a binary tensor of the same shape as the target sequence — 0 for
  question tokens, 1 for answer tokens. The loss function then only accumulates
  error on the positions where mask=1:

  loss = cross_entropy(logits, targets, reduction='none')  # per-token loss
  loss = (loss * mask).sum() / mask.sum()                  # average over answer
   tokens only

  Without the mask the model would spend most of its capacity learning to
  reproduce the "Question: ... Answer:" template, since that's the majority of
  tokens in each sample. With the mask it focuses entirely on learning what
  comes after "Answer:".

❯ An interesting observation. I ran the wikipedia training for 150k, then       
  started another training run to reach 550k. When the second run started the   
  loss jumped from about 2.4 to about 2.6. It then never fully recovered to     
  reach that minimum, and the progress was much slower. Is this a result of the 
   learning rate curve? I set the maximum number of steps to 1M for the second  
  part.                                                                         

● Yes, almost certainly the learning rate schedule. Here's what happened:

  The first run (0→150k steps):
  The cosine schedule ran from lr=3e-4 down to lr_min=0 over 150k steps. By step
   150k the LR had decayed to essentially zero — the model had converged tightly
   into a minimum.

  The second run (150k→550k, configured for 1M steps):
  When you resumed, the LR schedule restarted from scratch — warmup from 0 to
  3e-4 over 1000 steps, then cosine decay planned over 1M steps. So at step 1 of
   the second run, the LR jumped back up to near 3e-4.

  That sudden high LR kicked the model out of the good minimum it had settled
  into at the end of the first run — hence the loss jumping from 2.4 to 2.6. It
  then spent a long time slowly recovering because the cosine schedule is now
  decaying very gradually over 1M steps, keeping the LR relatively high for much
   longer than the first run did.

  The fix for future runs:

  Either:
  1. Set --max-steps to the total combined budget from the start — if you plan
  to train to 550k, configure --max-steps 550k on the first run even if you stop
   it early. The checkpoint stores the step number, so resuming will correctly
  continue the cosine decay from wherever it left off.
  2. Set --warmup-steps 0 when resuming — avoids the LR ramp at least, though
  the LR level itself will still be wrong relative to where the schedule should
  be.

  The root cause is that the LR schedule has no memory of the previous run — it
  only knows the current step number relative to max_steps.
