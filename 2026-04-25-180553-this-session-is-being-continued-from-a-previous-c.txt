  fires a warning: "the sequence you just tokenized is longer than the model can
   process."

   that was a previous attempt to silence this warning by raising the threshold.
   Shakespeare complete has 1,686,450 tokens, which still exceeds 1,000,000, so
  the warning fires anyway.

  Why this warning doesn't matter for us: We are not feeding these tokens to
  GPT-2. We're using GPT-2's tokenizer purely as a vocabulary — it converts
  characters to integer IDs using GPT-2's BPE rules. We don't care about GPT-2's
   1024-position limit because our own model handles its own positional
  encodings, and TokenDataset slices the token stream into 1024-token chunks
  before anything sees them. The tokenizer is just a lookup table at this point.
   Setting model_max_length = 10_000_000 silences the warning without any other
  effect.

  ---
  3. The RAM fix — explained

  The original problem: stride = 1

  TokenDataset.__len__ previously returned len(tokens) - context_length, which
  is the number of all possible starting positions. For Wikipedia:

  8.6 GB file ÷ 2 bytes/token = 4,300,000,000 tokens
  __len__ = 4,300,000,000 - 1024 ≈ 4.3 billion samples

  Every byte offset is a valid starting point: sample 0 starts at token 0,
  sample 1 starts at token 1, sample 2 at token 2, etc. This gives maximum
  coverage but means adjacent samples overlap almost entirely (1023 of 1024
  tokens shared).

  When you create a DataLoader with shuffle=True, PyTorch generates a random 
  permutation of all sample indices before the first batch. For 4.3 billion
  samples:

  4,300,000,000 indices × 8 bytes (int64) = 34 GB RAM

  This allocation happens before any training, which is why the machine ran out
  of RAM during "startup."

  The fix: stride = context_length (non-overlapping chunks)

  The new __len__ is (len(tokens) - 1) // context_length. Instead of every
  starting position, only positions 0, 1024, 2048, 3072, ... are valid:

  4,300,000,000 tokens ÷ 1024 = ~4,200,000 samples
  4,200,000 indices × 8 bytes = 34 MB RAM

  __getitem__ now computes start = idx * context_length to find each chunk.

  The tradeoff is negligible for training: with stride=1, consecutive samples
  share 99.9% of their tokens (almost no new information per step). With
  stride=1024, each sample is completely fresh data. Language model training
  always uses non-overlapping or large-stride chunks in practice — it's the
  standard.

❯ Re 2, we have a position limit of 1024 as well - can our model successfully   
  train itself on the input data given that we have this hard limit of 1024     
  context length/position embedding?                                            

● Yes — and this is by design. Let me walk through exactly how it works.

  The token stream is one long sequence:
  [tok_0, tok_1, tok_2, ..., tok_4,300,000,000]

  TokenDataset slices this into non-overlapping chunks of exactly 1024 tokens.
  Each chunk becomes one training sample. The model never sees more than 1024
  tokens at once — it's not that long sequences get truncated, it's that they're
   never presented to the model in the first place.

  What the model learns from each chunk:

  A single chunk [t_0, t_1, ..., t_1023] generates 1024 training examples
  simultaneously due to the causal masking. At position 0, the model predicts
  t_1 having seen only t_0. At position 1, it predicts t_2 having seen t_0, t_1.
   At position 512, it predicts t_513 having seen 512 tokens of context. At
  position 1023, it has the full 1024-token context window. All 1024 predictions
   happen in one forward pass — that's why the loss is cross_entropy over all
  B×T positions, not just the last one.

  So what can't the model do?

  It cannot learn dependencies that span more than 1024 tokens. If a plot detail
   introduced at token 0 of a document becomes relevant at token 2000, the model
   will never see both in the same context window and cannot learn that
  relationship. This is the actual constraint.

  For Shakespeare or Wikipedia this is usually fine — most meaningful linguistic
   structure (sentences, paragraphs, arguments) fits within 1024 tokens. For
  very long documents it matters more, which is why modern models (GPT-4,
  Gemini) extend context to 8K–1M tokens via RoPE and architectural changes.
  RoPE is on our roadmap.

  The tokenizer warning was a red herring:

  The warning was about feeding 1.68M tokens to the tokenizer in one call.
  That's a tokenizer API concern — it processes the text fine, it just warns
  that the resulting token sequence is longer than GPT-2 can process as a model.
   None of those 1.68M tokens ever hit our model as a unit.

  learned vector to each position (0, 1, 2, ..., 1023). So the token "be" at

  it does at position 800 of a long document — the model sees them as different
  things.

  attention scores, so the model reasons about "this token is 5 positions before
   that one" rather than "this token is at absolute position 3." That means a
  short prompt at inference time is no longer out-of-distribution compared to
  training — "To be" followed by ", or not to be" still has the same relative
  positions regardless of where it appears in the context window.

  training and inference. It also enables length generalisation — the model can
  handle sequences longer than it was trained on, at least partially. That's why
   every modern architecture (LLaMA, Gemma, Mistral) uses RoPE. It's the next
  meaningful upgrade to make after you get Wikipedia training working.

❯ How does the cosine learning rate work and why is this particular functional  
  form of learning rate used?                                                   

● The functional form

  Our implementation:

  progress = (step - warmup_steps) / (max_steps - warmup_steps)  # 0 → 1
  lr = lr_min + 0.5 * (lr - lr_min) * (1 + cos(π * progress))

  At progress=0: cos(0) = 1, so lr = lr_min + (lr - lr_min) = lr — starts at the
   peak.
  At progress=1: cos(π) = -1, so lr = lr_min + 0 — ends at the floor.
  At progress=0.5: cos(π/2) = 0, so lr = midpoint — halfway through.

  The derivative of cos is -sin, which is zero at both endpoints and maximum in
  the middle. So the LR changes most rapidly in the middle of training and
  barely changes at the start and end. This gives a slow-start, slow-finish
  decay profile.

  ---
  Why decay the LR at all

  Think of the loss surface as a hilly landscape and the optimizer as a ball
  rolling downhill. A high LR gives the ball a lot of momentum — it covers
  ground quickly and escapes shallow local minima, but when it reaches a valley
  it overshoots and oscillates rather than settling. A low LR takes tiny steps
  and converges precisely, but gets stuck easily and makes slow progress from a
  bad starting point.

  The schedule exploits both: use a high LR early to find a good region of the
  loss landscape, then lower it so the model can converge tightly into a
  minimum. Constant LR never fully converges — it oscillates forever around the
  optimum. Decaying LR to near-zero lets the model settle to its sharpest, most
  confident state, which is exactly what you want for the memorization
  experiment.

  ---
  Why cosine specifically, not linear or step decay

  Step decay (drop LR by 10× every N steps) is discontinuous — the sudden jump
  destabilises training and the timing requires manual tuning.

  Linear decay reduces at a constant absolute rate. The problem: late in
  training, when LR is already small (say 1e-5), a linear decrement of 1e-5 per
  1000 steps is a 100% change relative to the current value — too aggressive
  near convergence.

  Exponential decay reduces multiplicatively, which is smooth but never reaches
  zero — it asymptotes. Also constant in log-space, which doesn't match how loss
   landscapes behave near minima.

  Cosine is smooth, bounded, and has the natural property that the rate of
  change is proportional to sin(π·progress) — small at both ends, large in the
  middle. Early in training the model is far from any minimum and changes are
  cheap; middle of training is the productive descent phase where fast LR change
   is fine; late in training the model is near a minimum and you want gentle,
  stable convergence. Cosine matches this structure without requiring you to
  hand-tune any inflection points.

  It was popularised by Loshchilov & Hutter (2017) in SGDR: Stochastic Gradient 
  Descent with Warm Restarts, where they showed it consistently outperformed
  step schedules empirically. It's now the default in almost every serious
  training run.

  ---
  Why the warmup

  At step 0, all weights are random initialisation. The first few gradient
  signals can be extremely large and noisy — acting on them with a full-sized LR
   can send weights far in a bad direction before the optimiser has any stable
  estimate of the loss surface.

  For AdamW specifically there's a second reason: Adam maintains running
  estimates of the gradient mean (m) and gradient variance (v), both initialised
   to zero. These exponential moving averages need several hundred steps to warm
   up to accurate estimates (β1=0.9 means ~10 steps, β2=0.999 means ~1000
  steps). During warmup, the v estimate is still near zero, making the effective
   step size m/√v artificially large and noisy. Starting with a tiny LR limits
  the damage from these unreliable early updates.

❯ If step < warmup_steps, can't progress be negative?                           

● Yes. If step < warmup_steps, the warmup branch returns early so progress is
  never computed for those steps — but the cosine branch has no guard and would
  produce a negative progress if called for a step before warmup ends.

  Looking at the code:

  def get_lr(step, cfg):
      if step < cfg.warmup_steps:
          return cfg.lr * step / max(1, cfg.warmup_steps)
      progress = (step - cfg.warmup_steps) / max(1, cfg.max_steps -
  cfg.warmup_steps)
      return cfg.lr_min + 0.5 * (cfg.lr - cfg.lr_min) * (1 + math.cos(math.pi *
  progress))

  The return in the warmup branch means the cosine line is only reached when
  step >= warmup_steps, so step - warmup_steps >= 0 and progress is always in
  [0, 1]. The early return is doing the guarding implicitly.

  It's a reasonable point that this could be made more explicit — some
  implementations clamp progress just to be defensive:

  progress = max(0.0, (step - cfg.warmup_steps) / max(1, cfg.max_steps -
  cfg.warmup_steps))

  But the early return makes it correct as written.

● Based on the roadmap in the README and what's been discussed, the most
  impactful next steps in rough priority order:

  Architecture upgrades (each swaps one component so the effect can be
  isolated):
  1. RoPE — replaces learned absolute positional embeddings with relative
  position encoding inside attention. Fixes the inference position-mismatch
  problem we observed, and enables length generalisation. This is the most
  motivated upgrade given the memorization experiments.
  2. RMSNorm — replaces LayerNorm. Simpler (no mean subtraction), slightly
  faster, used by every modern architecture.
  3. SwiGLU — replaces GELU in the FFN. Used by LLaMA/Gemma/Mistral; empirically
   stronger than GELU.

  Training:
  4. Wikipedia training run — the model has only been trained to convergence on
  Shakespeare. A proper Wikipedia run is the main pending experiment.
  5. Gradient accumulation — allows an effectively larger batch size than fits
  in VRAM by accumulating gradients over multiple forward passes before stepping
   the optimizer.

  Scale:
  6. GPT-2 small (d=768, 12 layers, 12 heads, ~117M params) — after Wikipedia
  validates the pipeline, scaling up is the natural next step.

  The natural order would be: implement RoPE + RMSNorm + SwiGLU first (since

❯ After training for 150k steps, the model reaches a loss of about 3, and seems 
   to consistently oscillate around this value. The evaluation reports a        
  similar value. Does this suggest the model is too small to be able to learn   
  all of the information contained within the Wikipedia dataset?                

● Not necessarily — a val loss tracking train loss closely is actually good
  news, it means the model is genuinely learning rather than memorizing. A loss
  of ~3 corresponds to perplexity of e³ ≈ 20, meaning the model is on average
  choosing between ~20 plausible next tokens at each position, which is
  reasonable language modelling behaviour.

  The oscillation at constant LR is expected — cosine decay hasn't brought the
  LR down much yet at 150k out of 500k steps. The model is still in the high-LR
  exploration phase. You'd expect it to continue improving as the LR decays,
  with the sharpest convergence happening in the final third of training.

  A loss floor around 3 for a 25M parameter model on Wikipedia is plausible but
  not a hard limit — GPT-2 small (117M params, same architecture family) reaches
   ~3.1 on WebText. The main factors that would push it lower are:

  1. More steps — the model may simply not have seen enough of the dataset yet
  2. More parameters — larger d_model and more layers can represent more of
  Wikipedia's diversity
  3. Architecture upgrades — RoPE, SwiGLU, RMSNorm all improve the model's
  capacity for a given parameter count

  The fact that val loss ≈ train loss is the important signal: the model isn't
  hitting a memorization ceiling, it's genuinely bottlenecked by capacity or
  training budget. Worth letting it run to 500k steps and seeing where the
  cosine decay lands it before drawing conclusions.

❯ If I run 2 training runs in a row, does the training for the second run start 
   using data from the beginning of the training dataset, or from the end?      

● From the beginning. Each run creates a fresh DataLoader and
  iter(train_loader), which shuffles the dataset independently from scratch.
  There's no state carried over between runs.

  Within a single run, if training exhausts all samples in the dataset (one full
   epoch), the StopIteration handler in the loop creates a new iterator, which
  reshuffles and starts from the beginning again.

❯ If each run is shuffled, then that means running two training runs of 30k     
  still results in useful training?                                             
● Yes. Because the shuffle is random and independent each run, the model sees a
  different ordering of samples in the second run — effectively continuing to
  learn from new arrangements of the data rather than repeating the exact same
  sequence. It's equivalent to running one long run from the model's learning
  perspective, minus the overhead of restarting.

  The only thing lost between runs is the optimizer's momentum state (m and v in
   AdamW), since that isn't carried over when starting fresh. This means the
  second run's first few steps are slightly less efficient while the momentum
  terms rebuild. But this is a minor effect — the LR warmup handles it.
